# 花束系统重设方案 v0.4

## 为什么重设

现有方向已经证明三个问题：

- 从整张资产板粗裁花材，进入 Cocos 后会像素材拼贴。
- 用少数单枝素材和简单槽位无法稳定形成饱满花束。
- 程序多边形包装纸很难变漂亮，继续微调是在错误方向上消耗时间。

新的核心判断是：

> 玩家负责选择与投入，系统负责花艺整理。

首版不追求真实自由插花。玩家拖入一份花材，系统把它落入模板中的花艺角色组，并生成多个内部图层，保证成品稳定漂亮。

## 视觉目标

首版追求清晰、明亮、柔和、带线稿的高精度 2D 插画风。

关键特征：

- 线稿清楚，手机小屏可读。
- 色彩丰富但不过分写实，花朵有粉、桃、蓝、白、绿的明确层次。
- 花束整体呈上宽下窄的扇形。
- 花材密度高，避免稀疏空洞。
- 包装为浅牛皮纸或暖米色纸，结构简单，边缘线条明确。
- 蓝白条纹丝带和小吊牌可作为首束花记忆点。
- UI 和背景只能衬托花束，不抢视觉中心。

## 首束花内容

游戏内仍维持 4 类可种植花材，避免扩大新手复杂度：

| 游戏角色 | 当前内容名 | 概念图对应 | 用途 |
| --- | --- | --- | --- |
| 主花 | 粉色大丽花 | 粉色芍药/大丽花视觉 | 最大视觉焦点 |
| 配花 | 香槟色洋牡丹 | 桃橙玫瑰/洋牡丹 | 补体积和暖色 |
| 线条花 | 蓝色飞燕草 | 蓝色飞燕草/小蓝花枝 | 拉高轮廓 |
| 填充花 | 白色小雏菊 | 小白花与雏菊 | 填缝和增加精致感 |

为了视觉成立，订单还会自动提供：

- 绿叶组。
- 包装纸组。
- 丝带。
- 小吊牌。
- 少量装饰星光或完成特效。

这些自动层不消耗玩家库存。

## 资产规格

### 花材资产

每类花材不再只做 2 个孤立素材，而是做一个可插花的素材组。

P0 最小资产：

| 资产组 | 最小数量 | 建议尺寸 | 说明 |
| --- | ---: | --- | --- |
| 主花大花头 | 3 | `512 x 512` | 左、中、右不同朝向 |
| 主花小花头或半遮挡 | 2 | `384 x 384` | 用于前后层次 |
| 配花花头 | 5 | `384 x 384` | 桃橙、香槟、浅粉变化 |
| 线条花枝 | 4 | `512 x 768` | 左倾、右倾、背后高枝 |
| 填充小花束 | 6 | `384 x 384` | 白色小花、雏菊、小枝 |
| 圆叶/桉叶 | 4 | `384 x 512` | 外侧轮廓与中层遮挡 |
| 蕨叶/尖叶 | 4 | `384 x 512` | 底部和侧边轮廓 |

要求：

- 透明 PNG。
- 统一线稿粗细、光照方向和色彩饱和度。
- 花材必须是插花枝材，不要画成完整盆栽或单株植物。
- 锚点以花茎底部或枝材插入点为逻辑中心。
- 缩到游戏内 80 至 160 像素高度仍能识别。

### 包装资产

包装不再程序绘制，改为分层图片。

P0 最小资产：

| 图层 | 建议尺寸 | 层级 |
| --- | --- | --- |
| 后包装纸左翼 | `768 x 1024` | 花材后方 |
| 后包装纸右翼 | `768 x 1024` | 花材后方 |
| 后包装纸中片 | `768 x 1024` | 花材后方 |
| 前包装纸左折 | `768 x 768` | 花材前方 |
| 前包装纸右折 | `768 x 768` | 花材前方 |
| 前包装纸中片 | `768 x 768` | 花茎前方 |
| 蓝白条纹丝带 | `768 x 384` | 最前方 |
| 小吊牌 | `256 x 384` | 丝带旁 |

要求：

- 包装要遮住大部分花茎底部。
- 包装轮廓必须和花束上宽下窄结构一致。
- 包装纸不追求复杂写实，优先清楚、干净、有手绘质感。

## 花束模板

首束花使用固定模板，不再让花材在画布上自由乱摆。

模板逻辑层级：

```text
后包装纸
→ 背景绿叶和高枝
→ 线条花
→ 配花后层
→ 主花后层
→ 主花前层
→ 填充小花
→ 前景绿叶
→ 前包装纸
→ 丝带和吊牌
→ 完成特效
```

玩家拖入一份花材后，系统填入对应角色的下一个可用槽位。一个玩家投放动作可以生成多个内部图层，但必须满足：

- 库存只消耗 1 份。
- UI 反馈只表现为放入 1 份花材。
- 交付页继续复用同一份模板状态，不替换成固定英雄图。

## DIY 交互

首版 DIY 目标从“自由摆放”改为“参与式完成漂亮模板”。

玩家可做：

- 把四类花材拖入花束。
- 观察花束逐步变丰满。
- 在完成后获得整理动画和交付卡片。

系统负责：

- 自动选择该花材的变体。
- 自动落位、缩放、旋转和遮挡。
- 自动补绿叶、包装、丝带和吊牌。
- 保证最终成品接近目标图的轮廓和密度。

首版暂不做：

- 玩家手动旋转、缩放单枝花。
- 任意拖动成品中的花材。
- 多套包装选择。
- 复杂评分算法。

## 开发顺序

1. 生成一张首束花资产板，风格对齐 Gemini 概念图。
2. 人工筛选出可拆分资产，并整理资产命名。
3. 生成透明 PNG 分层资产。
4. 建立 `bouquet-template-v04` 配置，定义每个图层的位置、缩放、角度和深度。
5. Cocos 中写一个模板渲染器，按配置渲染花束。
6. DIY 页面用模板状态替代当前散点槽位。
7. 交付页复用同一份模板状态。
8. 在手机尺寸下做截图对比，确认第一眼美感。

## 验收标准

v0.4 只有一个核心验收：

> 新手按引导完成第一束花后，玩家愿意说“这束花是好看的”。

具体标准：

- 花束整体轮廓接近 Gemini 概念图的饱满扇形。
- 包装纸不是程序多边形感。
- 花材不再像孤立贴图。
- 主花、配花、线条花、填充花在小屏下都能读出来。
- 花材根部被包装自然收住。
- DIY 页和交付页展示的是同一束由玩家完成的花。
- 真实浏览器 430 x 760 预览截图第一眼明显优于旧散点版本。

## 风险

- AI 资产容易风格不一致，需要先做资产板，再拆分，不要一张一张孤立生成。
- 完全透明切图可能有边缘残留，需要本地检查。
- 花束模板会降低自由度，但能显著提高首版成品质量。
- 如果模板仍不够漂亮，应继续调资产和模板，不应回头扩大玩法。
