# 花束系统重设方案 v0.5

## 结论

v0.4 的“重新生成大量分层资产 + 包装分层 + 模板式插花”方向仍然正确，但当前产出在 Cocos 里依然像旧素材拼贴，已经不适合作为下一步继续堆系统。

v0.5 先及时止损，只做一个更窄的花束制作验证：

参考图：

`视觉验证/GPT-第一轮/风格素材/开个花店素材/Gemini_Generated_Image_9481bn9481bn9481.png`

执行方式：

- 使用用户黑框标注的 4 个花头作为唯一可拖动花材。
- 从原图中精确抠出这 4 个花头，导出透明 PNG。
- 同步生成 4 个透明白色花朵骨架 PNG，用作未放入槽位提示。
- 将扣掉花头后的剩余画面作为 `template-base`。
- 玩家拖入花头后，系统只做固定位置、大小和深度还原，不再自由摆放。
- 本轮先验证“玩家通过少量动作复原一束漂亮花”是否成立。

这张图只作为本地审美和交互验证素材。公开版本仍必须使用可授权或重新生成、检查后的正式资产。

## 为什么再次收窄

现有方向已经证明三个问题：

- 从整张资产板粗裁花材，进入 Cocos 后会像素材拼贴。
- 用少数单枝素材和简单槽位无法稳定形成饱满花束。
- 程序多边形包装纸很难变漂亮，继续微调是在错误方向上消耗时间。

新的核心判断保持不变：

> 玩家负责选择与投入，系统负责花艺整理。

但当前阶段不再尝试“一次性拆出完整花材体系”。我们先用原型图本身做最小还原实验，确认 Cocos 中的拖拽、遮挡和最终观感可行，再决定是否投入正式资产生产。

## 视觉目标

首版追求清晰、明亮、柔和、带线稿的高精度 2D 插画风。

关键特征：

- 线稿清楚，手机小屏可读。
- 色彩丰富但不过分写实，花朵有粉、桃、蓝、白、绿的明确层次。
- 花束整体呈上宽下窄的扇形。
- 花材密度高，避免稀疏空洞。
- 包装为浅牛皮纸或暖米色纸，结构简单，边缘线条明确。
- 蓝白条纹丝带和小吊牌可作为首束花记忆点。
- UI 和背景只能衬托花束，不抢视觉中心。

## 当前 cutout-v01 内容

运行时只包含：

- `template-base.png`：扣除 4 个花头后的花束底图。
- `hero-left.png`：左侧最大粉色花头。
- `pink-upper-right.png`：右上粉色花头。
- `pink-middle-right.png`：右中粉色花头。
- `peach-lower-left.png`：下方桃橙花头。
- `*-skeleton.png`：对应花头的透明白色骨架占位。

不再包含：

- 旧版四类花材素材卡。
- 程序绘制包装纸。
- 多图层包装资产。
- 30 至 50 层完整插花模板。
- 种植、收割、订单奖励 UI。

## 资产规格

### cutout-v01 最小资产

资源目录：

```text
assets/resources/art/bouquet-cutout-v01/
```

文件：

| 文件 | 用途 |
| --- | --- |
| `template-base.png` | 静态花束底图，已经扣除 4 个可拖动花头 |
| `hero-left.png` | 左侧最大粉色花头 |
| `pink-upper-right.png` | 右上粉色花头 |
| `pink-middle-right.png` | 右中粉色花头 |
| `peach-lower-left.png` | 下方桃橙花头 |
| `*-skeleton.png` | 未放入时的透明白色花朵骨架占位 |

要求：

- 4 个花头必须是透明 PNG，不带黑框、底色块或紫红描边残留。
- 4 个骨架占位必须是透明 PNG，不再使用矩形框、半透明彩色花头或白色补丁。
- 花头在目标位置使用原图尺寸比例，不再人为重新设计角度。
- 用 `depth` 控制前后遮挡顺序。
- 验证图保存在 `视觉验证/第七轮-原图抠花验证/`。

## 运行时模板

首版验证页只保留一个固定模板：

```text
template-base
→ 透明白色花朵骨架占位
→ 玩家拖动 4 个花头
→ 花头吸附到固定坐标
→ 按 depth 重新渲染，复原原型图局部
```

本轮不做自由插花，不做包装选择，也不把交付页替换成一张固定英雄图。

## 后续正式资产方向

cutout-v01 如果在手机上看起来成立，下一步才恢复正式资产生产：

- 重新生成可商用、可授权的完整花束资产。
- 将当前 4 花头固定槽位扩展为花艺角色槽位。
- 包装重新做成清晰分层资产。
- 交付页复用玩家完成的同一束花。
- 再接回种植、收割、订单和奖励闭环。

## 开发顺序

1. 用标注图定位 4 个黑框在原图中的坐标。
2. 从原图抠出 4 个透明花头。
3. 生成扣除 4 个花头后的 `template-base`。
4. 生成还原预览图，确认 `template-base + 4 花头` 可复原。
5. Cocos 改为单一花束制作验证页。
6. 真实浏览器验证拖拽、吸附、层级和手机尺寸观感。

## 验收标准

v0.5 只有一个核心验收：

> 玩家拖完 4 个花头后，能明显看出这是原型图中那束漂亮花，而不是旧素材拼贴。

具体标准：

- 页面不再出现旧版主花素材、旧材料卡、旧程序包装纸。
- 4 个花头无黑框、无底色块、无明显紫红描边残留。
- 未放入状态不再出现矩形补丁，而是花朵骨架提示。
- 花头吸附后位置、大小和前后层级接近原图。
- 拖完 4 个花头后，最终图第一眼明显优于旧散点版本。
- 真实浏览器 `430 x 760` 预览可操作。

## 风险

- 当前使用原型图做本地验证，不是最终可发布资产。
- 扣掉花头后的模板在未完成状态会有空洞，需要用槽位提示遮住。
- 黑框标注区域较紧，花头边缘可能被截断；若手机预览明显不自然，需要重新标注更宽的抠图范围。
- 如果复原实验仍然不漂亮，问题就不在 Cocos，而在目标图拆分方式，需要重新选图或重新生成可拆分资产。
